# Stage 1 — 数据采集

> 把直播变成可分析的语料。

## 目标

为后续风格分析准备**多元 / 多情绪 / 多场景**的语料样本。最终交付:
- 14 场公开直播回放的音频文件(`.m4a`)
- 元信息(BVID / 标题 / 分 P 时长 / 上传日期)

## 数据源

- **平台**:Bilibili 公开直播回放(只下"我有权访问的"内容,符合主播本人立场和平台 ToS)
- **数量**:14 场,覆盖不同主题(闲聊 / 新年特别场 / 投稿处理 / 游戏 / 杂谈 / 歌回)
- **总时长**:约 50 小时
- **总分 P**:约 40 个(部分场是单 P,部分多 P)

## 工具

- **yt-dlp**(批量下载音频)
- 配置点:
  - `--format bestaudio[ext=m4a]` — 只要音频,不要视频流(节省体积 / 后续 whisper 也只需音频)
  - `--write-info-json` — 元信息存 JSON,后续可关联标题 / 上传日期 / 时长
  - `--no-overwrites` — 重复下载跳过
  - `--sleep-interval N` — 友善间隔,不要轰炸 CDN
  - `--cookies-from-browser chrome` — 用浏览器 cookie,确保有权访问

## 选择标准(哪 14 场)

**多元覆盖**比"按热度选"重要。挑选维度:
- **情绪光谱**:必须有"开心轻快""锐评毒舌""慵懒摆烂""真敏感时刻"等场次,不能只挑"出片"段
- **互动密度**:必须有 SC 密集场(粉丝投稿 / 念信)与 SC 稀疏场(纯杂谈)各几场
- **时间跨度**:跨越数月,避免某段时间情绪偏置影响归纳
- **特殊事件**:含至少 1-2 场涉及创伤话题(如坚果离世前后)的回放——L1 真敏感行为的样本来源

**反向避免**:
- 纯歌回场(信号密度低,信号几乎是歌词不是说话)
- 联动场(混入其他主播话语,字幕识别会乱)
- 极短(<30 分钟)的特别场(样本量太小,不稳定)

## 弹幕策略

**只对直播回放下载弹幕,不对粉丝切片下载**——这是经验教训:粉丝切片的"弹幕"是切片视频的事后回放弹幕,**不是奶绿原直播的实时弹幕**,对"复现直播现场互动"目标零价值。

**PII 处理**:弹幕用户名必须 hash 化或脱敏后再使用。任何中间产物公开前必须先脱敏。**本仓库不公开任何含 PII 的中间数据**。

## 数据卫生(Stage 1 的真实教训)

早期开发时,曾用一个测试 BVID(`BV14KDtBSEnS`,内容是"瓜保熟"梗段子 + 数码评测,非奶绿)做 yt-dlp / 字幕通道连通性测试。测试产物文件名为 `bili_ai_P{1,2,3}.json` 落在数据目录里——后续被 Stage 3 清洗脚本顺手处理,数据被静默污染。**直到 Stage 4 风格分析时跨场聚合,统计出现可疑词频(如"分体水冷"),才回溯发现**。

**教训**:
1. 任何测试性 ASR / 下载产物**必须放隔离目录**,不能混入正式数据流
2. 各阶段的输入应该有**命名白名单**(只接受符合规范的文件名,过滤其他)
3. 跨阶段流转的数据要有**抽样审查机制**

修复:在文本清洗阶段(Stage 3)加 `is_target_stem()` 白名单——只接受 `BV*` / `clip_BV*` / `*_BiliBili_*` 命名格式,其他一律 SKIP。

## 不公开的内容

- 14 场音频原文件(版权所有人:明前奶绿;研究使用 OK,公开传播侵权)
- yt-dlp 命令行配置 / 自动化脚本
- 元信息 JSON(含 BVID 列表 + 上传日期 + 时长,可能涉及隐私)

## 给后来者的话

- 自己复现请确保**下载的内容你有权访问**(账号订阅 / 公开直播回放 / 已购买等)
- 不要批量爬未公开内容
- 不要扫主播私号 / 私人投稿
- 14 场是经验值,样本太少(<10)会让 Stage 4 的"跨场稳定"判断不稳;样本太多(>30)反而稀释信号,因为不同时期情绪状态差异大
